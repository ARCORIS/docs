## Purpose

This section is the **audience-based entry layer** of ARCORIS documentation.

Use it when you want to begin from your role, goal, or immediate need rather than from the repository structure alone. Its job is to help readers choose the right documentation path quickly and move into the canonical documentation layers without unnecessary routing steps.

## When to use this section

Use **Start Here** if you are asking questions such as:

- where should I begin if I am new to ARCORIS;
- which documentation path fits my role;
- what should I read first as an evaluator, integrator, operator, or contributor;
- how do I reach the canonical system baseline without guessing;
- what is the practical difference between the main reader paths.

If you already know the documentation layer you need, you can go directly to the [Documentation Index](../index.md) or to the relevant section entry page.

## Path descriptions

### For Evaluators

Use [For Evaluators](for-evaluators.md) when your primary goal is to **understand ARCORIS as a system** before making a technical commitment.

This route is best when you want to determine:

- what ARCORIS is;
- what kind of problem space it addresses;
- what its goals and non-goals are;
- what kinds of use cases it is designed for;
- whether its architecture and operating model match the intended scenario.

### For Integrators

Use [For Integrators](for-integrators.md) when your primary goal is to **connect ARCORIS to surrounding systems or infrastructure**.

This route is best when you need to understand:

- where system boundaries are;
- which interfaces and protocols matter;
- how requests, workloads, and state move through the system;
- where extension and integration points are likely to matter;
- which conceptual and architectural material should come before interface-level work.

### For Operators

Use [For Operators](for-operators.md) when your primary goal is to **run, observe, secure, and support ARCORIS in practice**.

This route is best when you need to understand:

- runtime shape and operational concerns;
- observability and operational signals;
- security-facing deployment concerns;
- failure-oriented behavior and major recovery-related flows;
- which documentation layers matter before reading operational material in depth.

### For Contributors

Use [For Contributors](for-contributors.md) when your primary goal is to **understand the system baseline before making changes**.

This route is best when you need to know:

- which canonical documentation should be read first;
- how to avoid changing code or documentation without understanding system boundaries;
- how overview, concepts, and architecture fit together as a pre-change baseline;
- where to go next for contribution workflow, operating rules, community structures, and enhancement processes.
